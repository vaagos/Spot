# Функциональные требования

Требования описаны в формате **Use Case** для функций MVP. Каждый сценарий содержит ID, акторов, предусловия, основной поток, постусловия, альтернативные потоки, исключительные потоки и бизнес-правила.

## UC-1. Регистрация нового пользователя

| Раздел | Описание |
| --- | --- |
| **ID** | UC-1 |
| **Название** | Регистрация нового пользователя |
| **Акторы** | Пользователь, желающий создать новый аккаунт; База данных пользователей; SMS-сервис отправки кодов |
| **Предусловия** | Пользователь не авторизован в приложении; есть интернет-соединение |

### Основной поток

1. Пользователь открыл приложение.
2. Пользователь пролистал приветственный экран.
3. Система предлагает зарегистрироваться по номеру телефона или войти в аккаунт.
4. Пользователь нажимает «Зарегистрироваться по номеру телефона».
5. Система открывает экран ввода номера телефона.
6. Пользователь вводит номер телефона.
7. Система валидирует формат номера.
8. Пользователь нажимает «Получить код».
9. Система отправляет запрос в БД пользователей и проверяет, существует ли пользователь с таким номером.
10. Система открывает экран ввода кода подтверждения.
11. Система отправляет запрос в SMS-сервис для отправки кода.
12. Пользователь получает код в SMS.
13. Пользователь вводит код подтверждения.
14. Система валидирует код.
15. Система открывает экран создания пароля.
16. Пользователь вводит пароль.
17. Система валидирует пароль.
18. Система добавляет пользователя в БД.
19. Пользователь авторизуется и попадает в каталог маршрутов.

### Постусловия

**Успех:**

- Пользователь получает доступ к своему аккаунту.
- Аккаунт пользователя добавляется в БД.
- При повторном открытии приложения пользователь остаётся авторизован.

**Неудача:**

- Отображается сообщение об ошибке.
- Пользователь может повторить попытку или написать в поддержку.

### Альтернативный поток 1. Аккаунт уже зарегистрирован

**Точка входа:** шаг 7.

1. Система обнаруживает наличие аккаунта в БД.
2. Интерфейс отображает «Номер телефона уже используется».
3. Система предлагает перейти к экрану авторизации.
4. Пользователь переходит на экран входа.
5. Введённый ранее телефон подставляется автоматически.

### Альтернативный поток 2. Код не доходит из-за плохой связи

**Точка входа:** шаг 12.

1. Пользователь не получает код в SMS.
2. Система предлагает отправить код повторно.
3. Система ограничивает количество повторных отправок (повторная отправка через 3 минуты).
4. При повторной отправке предыдущий код становится недействителен.
5. Пользователь вводит новый код.

### Исключительные потоки

**Сервис отправки кодов недоступен.** Точка входа: шаг 11.

1. Система не получает ответ от SMS-сервиса.
2. Сообщение: «Не удалось отправить код. Попробуйте снова через некоторое время».
3. Пользователь может повторить попытку введения кода.

### Бизнес-правила

| ID | Правило |
| --- | --- |
| BR-01 | Для одного номера телефона — один аккаунт |
| BR-02 | Номер телефона должен соответствовать формату РФ |
| BR-03 | Код подтверждения действует 5 минут |
| BR-04 | Максимальное количество попыток ввода кода — 5 |
| BR-05 | После превышения количества попыток возможность создать аккаунт временно блокируется на 15 минут |
| BR-06 | Минимальная длина пароля — 6 символов |

## UC-2. Авторизация существующего пользователя

| Раздел | Описание |
| --- | --- |
| **ID** | UC-2 |
| **Название** | Авторизация существующего пользователя |
| **Акторы** | Пользователь, желающий войти в существующий аккаунт; База данных пользователей; SMS-сервис |
| **Предусловия** | Пользователь не авторизован в приложении; есть интернет-соединение |

### Основной поток

1. Пользователь открыл приложение.
2. Пользователь нажимает «Уже есть аккаунт».
3. Система открывает экран ввода номера телефона и пароля.
4. Пользователь вводит номер телефона.
5. Система валидирует формат номера.
6. Пользователь вводит пароль.
7. Пользователь нажимает «Войти в аккаунт».
8. Система отправляет запрос в БД и проверяет:
   - существует ли пользователь с таким номером;
   - подходит ли к этому номеру телефона пароль.
9. Пользователь авторизуется и попадает в каталог маршрутов.

### Постусловия

**Успех:**

- Пользователь получает доступ к своему аккаунту.
- При повторном открытии приложения пользователь остаётся авторизован.

**Неудача:**

- Отображается сообщение об ошибке.
- Пользователь может повторить попытку или написать в поддержку.

### Альтернативный поток 1. Пользователь забыл пароль

**Точка входа:** шаг 8.

1. Интерфейс отображает «Неверный номер телефона или пароль».
2. Пользователь понимает, что забыл пароль.
3. Система предлагает кнопку «Не помню пароль».
4. Пользователь нажимает «Не помню пароль».
5. Система открывает экран восстановления пароля.
6. Пользователь подтверждает номер телефона.
7. SMS-сервис отправляет код восстановления.
8. Пользователь вводит код.
9. Система открывает экран создания нового пароля.
10. Пользователь вводит новый пароль.
11. Система валидирует пароль.
12. Система обновляет пароль в БД.
13. Пользователь авторизуется и попадает в каталог маршрутов.

### Исключительные потоки

**Авторизация временно заблокирована.** Точка входа: шаг 8.

1. Система фиксирует максимальное количество попыток ввода пароля.
2. Система переводит сессию с этого устройства в статус временной блокировки.
3. Сообщение: «Слишком много попыток входа. Попробуйте через 15 минут».

### Бизнес-правила

| ID | Правило |
| --- | --- |
| BR-01 | Номер телефона должен соответствовать формату РФ |
| BR-02 | Максимальное количество попыток входа — 5 |
| BR-03 | После превышения количества попыток вход временно блокируется на 15 минут |
| BR-04 | После смены пароля все активные сессии пользователя аннулируются |
| BR-05 | Повторный код восстановления можно запросить через 30 минут |
| BR-06 | Минимальная длина пароля — 6 символов |

## UC-3. Каталог маршрутов

| Раздел | Описание |
| --- | --- |
| **ID** | UC-3 |
| **Название** | Каталог с маршрутами |
| **Акторы** | Авторизованный пользователь; База данных маршрутов; Сервис геолокации (для раздела «Поблизости») |
| **Предусловия** | Пользователь авторизован; пользователь открыл каталог маршрутов; есть интернет-соединение |

### Основной поток

1. Пользователь открыл каталог маршрутов.
2. Система запрашивает в БД список маршрутов и информацию о них.
3. Система отображает в каталоге:
   - три раздела: **Популярное** (по умолчанию), **Новое**, **Поблизости**;
   - в выбранном разделе — карточки маршрутов;
   - каждая карточка содержит обложку маршрута (фото), теги (время и длина маршрута), ник автора, название и краткое описание.
   При клике на карточку открывается страница маршрута.
4. Пользователь может листать каталог или нажать на понравившуюся карточку.

### Постусловия

**Успех:**

- Пользователь видит актуальные карточки маршрутов.
- Пользователь может переключаться между разделами.
- Пользователь может перейти к детальной странице маршрута.

**Неудача:**

- Отображается сообщение об ошибке загрузки маршрутов.
- Пользователь может повторить попытку или написать в поддержку.

### Альтернативный поток 1. Потеря интернет-соединения

**Точка входа:** шаг 2.

1. Система не получает ответ от БД.
2. Сообщение: «Проверьте подключение к интернету».
3. Пользователь может нажать кнопку «Повторить».

### Альтернативный поток 2. Пользователь не дал доступ к геолокации

**Точка входа:** шаг 3, при переходе в раздел «Поблизости».

1. Система запрашивает доступ к геолокации.
2. Пользователь отклоняет запрос.
3. Сообщение: «Для отображения маршрутов поблизости необходим доступ к геолокации».
4. Система предлагает перейти в настройки.

### Исключительные потоки

**Данные загрузились не полностью.** Точка входа: шаг 2.

1. Карточка отобразилась без изображения.
2. Отображается placeholder.
3. Ошибка логируется.
4. Пользователь может перезагрузить раздел или приложение.

### Бизнес-правила

| ID | Правило |
| --- | --- |
| BR-01 | Раздел «Популярное» сортируется по количеству прохождений за последние 30 дней |
| BR-02 | Раздел «Новое» сортируется по дате публикации (сначала новые) |
| BR-03 | Раздел «Поблизости» сортируется по расстоянию от текущего местоположения |
| BR-04 | Удалённые маршруты не отображаются в каталоге |
| BR-05 | При частичной ошибке карточка отображается с placeholder |
| BR-06 | При ошибке отображается сообщение «Проверьте подключение к интернету» и кнопка «Повторить» |

## UC-4. Страница маршрута

| Раздел | Описание |
| --- | --- |
| **ID** | UC-4 |
| **Название** | Страница маршрута |
| **Акторы** | Авторизованный пользователь; База данных маршрутов |
| **Предусловия** | Пользователь авторизован; маршрут в статусе «Опубликован»; пользователь открыл карточку в каталоге; есть интернет-соединение |

### Основной поток

1. Пользователь открыл каталог маршрутов.
2. Пользователь нажал на карточку понравившегося маршрута.
3. Система запрашивает данные у БД маршрутов.
4. Система открывает страницу маршрута.
5. Страница маршрута отображает:
   - фото-баннер маршрута;
   - теги с продолжительностью по времени и дистанции;
   - название маршрута;
   - описание маршрута;
   - последовательность точек;
   - краткое описание точек с фотографиями;
   - рекомендации по маршруту;
   - кнопку «Исследовать маршрут» с переходом к карте.
6. Пользователь может вернуться в каталог, пролистать страницу или нажать «Исследовать маршрут».

### Постусловия

**Успех:**

- Пользователь видит страницу маршрута.
- Пользователь может перейти к карте, нажав «Исследовать маршрут».

**Неудача:**

- Отображается сообщение об ошибке загрузки.
- Пользователь может повторить попытку или написать в поддержку.

### Альтернативные потоки

**Пользователь не дал доступ к геолокации.** Точка входа: шаг 6, при нажатии на «Исследовать маршрут».

1. Система запрашивает доступ к геолокации.
2. Пользователь отклоняет запрос.
3. Сообщение: «Для отображения маршрута необходим доступ к геолокации».
4. Система предлагает перейти в настройки.

### Исключительные потоки

**База данных с информацией о маршруте не отвечает.** Точка входа: шаг 3.

1. Система не получает ответ от БД маршрутов.
2. Сообщение: «Сервис не работает, попробуйте снова через некоторое время».
3. При восстановлении связи система автоматически обновляет страницу.

### Бизнес-правила

| ID | Правило |
| --- | --- |
| BR-01 | Маршрут доступен пользователю только в статусе «Опубликован» |
| BR-02 | Удалённый маршрут становится недоступным немедленно |
| BR-03 | Маршрут должен содержать: название, изображение для баннера, минимум 1 точку |
| BR-04 | Страница маршрута не может содержать запрещённый контент и нецензурную лексику |
| BR-05 | Максимальное количество точек в маршруте — 50 |

## UC-5. Карта с отображением маршрута

| Раздел | Описание |
| --- | --- |
| **ID** | UC-5 |
| **Название** | Карта с отображением маршрута |
| **Акторы** | Авторизованный пользователь; База данных маршрутов; Yandex Maps API |
| **Предусловия** | Пользователь авторизован; разрешение на доступ к геолокации получено; маршрут в статусе «Опубликован»; пользователь выбрал маршрут для прохождения; есть интернет-соединение |

### Основной поток

1. Пользователь нажал кнопку «Исследовать маршрут».
2. Система использует ранее полученные данные маршрута со страницы маршрута.
3. Система загружает карту через Yandex Maps API.
4. Система отображает на карте:
   - иконку пользователя в текущей позиции;
   - путь пешком до следующей точки маршрута;
   - все точки маршрута;
   - точки интерактивные — при нажатии открывается поп-ап страницы точки с фото, названием маршрута и точки, описанием;
   - трекинг прогресса («Вы прошли 3/6 локаций», «Осталось 48 минут», полоска прогресса);
   - кнопку «Закончить маршрут» с возможностью вернуться в каталог.
5. Система обновляет координаты каждые 3 секунды.
6. При обновлении иконка пользователя плавно перемещается в новую позицию.

### Постусловия

**Успех:**

- Пользователь видит карту маршрута.
- Текущая позиция пользователя корректно отображается.
- Пользователь может взаимодействовать с точками на карте.
- Текущая позиция меняется в зависимости от прохождения маршрута.
- Данные о завершении маршрута фиксируются в БД.

**Неудача:**

- Отображается сообщение об ошибке загрузки карты.
- Пользователь может повторить попытку или написать в поддержку.

### Альтернативный поток 1. Неточная геопозиция пользователя

**Точка входа:** шаг 4.

1. Сообщение «Неточная геопозиция».
2. Система предлагает отмечать пройденные точки вручную.
3. Пользователь нажимает «Точка пройдена» и переходит на следующую точку.

### Альтернативный поток 2. Пользователь открывает точку не по порядку

**Точка входа:** шаг 4.

1. Пользователь нажимает на точку маршрута на карте.
2. Система отображает информацию о точке, но не меняет прогресс.
3. Пользователь может свернуть попап и продолжить маршрут или посмотреть другую точку.

### Исключительные потоки

**Ошибка расчёта маршрута.** Точка входа: шаг 4.

1. Yandex Maps API не может построить пешеходный маршрут.
2. Отображается прямая линия до точки.
3. Сообщение: «Не удалось построить маршрут, попробуйте позже».

### Бизнес-правила

| ID | Правило |
| --- | --- |
| BR-01 | Если пользователь завершил маршрут, дальнейшее обновление геопозиции прекращается |
| BR-02 | Карта должна отображать только точки активного маршрута |
| BR-03 | Система обновляет координаты каждые 3 секунды |
