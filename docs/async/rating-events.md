# Асинхронное взаимодействие: оценка маршрута

## Сценарий

Когда пользователь ставит оценку маршруту (1–5 звёзд), событие оценки отправляется в очередь для асинхронного пересчёта среднего рейтинга маршрута.

## Описание процесса

1. Пользователь нажимает «Оценить» на странице пройденного маршрута.
2. `rating-service` валидирует оценку, сохраняет её в PostgreSQL (таблица `ratings`).
3. `rating-service` публикует событие `RouteRatingChanged` в Kafka-топик.
4. Пользователь сразу видит «Оценка принята».
5. `aggregation-service` читает события из топика, пересчитывает средний рейтинг и количество оценок, обновляет их в таблице `routes`.
6. Параллельно `analytics-service` читает те же события для аналитики (популярность маршрутов по городам, динамика рейтинга во времени).

## Почему это асинхронное взаимодействие

| Причина | Объяснение |
| --- | --- |
| **Нет необходимости ждать результат** | Пользователю важно увидеть «Оценка принята», а не новый средний рейтинг. Обновление среднего может занять время — это не повлияет на UX |
| **Пересчёт может быть долгим** | Средний рейтинг считается по всем оценкам маршрута. Для популярных маршрутов с тысячами оценок это медленная операция. Делать её синхронно — значит блокировать запрос пользователя |
| **Бизнес-данные для продолжения не нужны** | Пользователь не продолжает флоу на основе нового рейтинга |

## Выбор технологии: Apache Kafka

### Обоснование

| Критерий | Объяснение |
| --- | --- |
| **Event sourcing** | Kafka — распределённый брокер, построенный вокруг концепции лога событий. Важно сохранять историю всех оценок не только для пересчёта рейтинга, но и для аналитики (изменение рейтинга маршрута во времени, динамика активности пользователей). RabbitMQ для этого не подходит: после `ack` сообщение удаляется |
| **Порядок сообщений** | Для корректного пересчёта рейтинга события по одному маршруту должны обрабатываться в порядке их создания. Kafka гарантирует порядок внутри партиции |
| **At-least-once + идемпотентность** | Консьюмер использует `messageId` для защиты от дублей. При повторной обработке одного события средний рейтинг не посчитается дважды |
| **Batch-обработка** | Kafka позволяет вычитывать сообщения батчами и делать массовые обновления. `aggregation-service` может брать 100 оценок за раз и одним запросом обновлять рейтинги 100 маршрутов, что разгружает PostgreSQL |

### Платформа и стек

- **Backend Spot** (Java/Kotlin или Python) — имеет зрелые Kafka-клиенты (`kafka-clients`, `confluent-kafka-python`).
- **DWH:** в инфраструктуре уже планируется ClickHouse, Kafka логично встаёт перед ним как источник данных.
- **Schema Registry:** Confluent или Apicurio для Avro-схем.
